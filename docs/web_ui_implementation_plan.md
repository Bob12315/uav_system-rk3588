# Web UI Implementation Plan

## 1. Goal

Add a browser-based operator UI to the UAV application for both supported
deployment modes:

- x86 host running YOLO locally and reaching the flight controller through a
  telemetry/data link.
- RK3588 host running the RKNN YOLO backend and reaching the flight controller
  over its local wired link.

The browser is used from a laptop on the same local network. The x86 and
RK3588 deployments must expose the same Web UI behavior; only their YOLO
inference backend and platform defaults differ.

## 2. Confirmed Product Requirements

### Control Page

- Display the annotated video generated by `yolo_app`.
- Display current detections and current locked target.
- Lock a target by clicking its bounding box in the video.
- Lock a target by choosing it in a detection list.
- Provide `next`, `previous`, and `unlock` target actions.
- Provide explicit flight controls:
  - change mode to `GUIDED`
  - arm
  - takeoff using an operator-entered altitude
  - stop automatic sending (`control send off`)
  - return-to-launch (`mode RTL`)
  - land
  - disarm with a strong confirmation
- Provide SITL/REAL active-source selection. Switching source must disable
  automatic sending and clear continuous control first.
- Dynamically list missions; new missions must not require front-end changes.
- Provide mission switch, start, reset, and current stage/hold state.
- Provide a command input with completion and command history.
- Separate operator actions from system output:
  - operator audit pane: persisted commands/button/video/config/service actions
  - compact system state pane: current status and latest control output
  - event pane: significant state changes and operation results

### Configuration Page

- Edit only project-approved YAML files:
  - `config/app.yaml`
  - `config/telemetry.yaml`
  - `yolo_app/config.yaml`
  - `missions/*/config.yaml`
- Support raw YAML editing and diff preview.
- Permanently save edits after YAML validation.
- Keep one previous copy as `<file>.bak`, and restore it on demand.
- For the active mission, offer save-and-apply:
  - disable command sending
  - clear pending continuous control
  - save the YAML
  - hot reload mission controller parameters
  - keep command sending disabled
- For telemetry configuration, offer save-and-reconnect:
  - disable command sending and clear continuous control
  - save the YAML
  - rebuild the telemetry link with the saved configuration
  - keep command sending disabled
- For YOLO configuration, offer save-and-restart YOLO.
- For app configuration, offer save-and-restart app; the browser reconnects
  after the service resumes.

### Deployment and Access

- No login/authentication in version one; operation is restricted by network
  deployment practice to a trusted LAN.
- Both terminal and Web UIs remain supported and are independently enabled in
  `config/app.yaml`:

```yaml
ui:
  web_enabled: true
  terminal_enabled: false
  web_host: "0.0.0.0"
  web_port: 8080
  audit_log_path: "logs/web_ui/audit.jsonl"
```

- YOLO local display and web stream are independently configured in
  `yolo_app/config.yaml`:

```yaml
display:
  local_window_enabled: false
  fullscreen: false

web_stream:
  enabled: true
  host: "0.0.0.0"
  port: 8081
  jpeg_quality: 75
  max_fps: 20
```

## 3. Architectural Boundary

```text
Browser
  | HTTP / WebSocket
  v
app.main + web_ui
  | existing UI command dispatcher / SystemRunner runtime API
  v
missions + telemetry_link + fusion

Browser <--- MJPEG annotated frames --- yolo_app
app <--- UDP target + scene JSON ------- yolo_app
app ---- UDP target commands ----------> yolo_app
```

### `web_ui/`

- Hosts the HTML/CSS/JavaScript application with FastAPI.
- Serves API endpoints and WebSocket snapshots/events.
- Delegates commands to the existing `uav_ui` command handler.
- Uses a whitelisted YAML configuration store.
- Persists operator audit entries as JSON lines.
- Does not build MAVLink messages and does not perform control computation.

### `app/`

- Owns runtime state, mission control, telemetry reconnect, safe send-disable
  actions, mission configuration reload, and service restart requests.
- Provides stable runtime snapshots for Web UI presentation.

### `yolo_app/`

- Continues to own video capture, inference, target selection, annotation, and
  the existing UDP contracts.
- Adds an MJPEG publisher for its already annotated frames.
- Maintains identical output behavior across the PyTorch/x86 and RKNN/RK3588
  implementations.

The video element reads the configured YOLO stream URL:

```text
http://<device-address>:8081/video/yolo.mjpeg
```

## 5. Safety Rules

- `executor.send_commands` remains false by default.
- Any action that changes the control environment first invokes the same
  runtime safety operation:
  - set `SEND=OFF`
  - clear pending continuous control output
  - record an operator and system event
- Safety operation is mandatory before:
  - mission configuration hot apply
  - telemetry reconnect
  - SITL/REAL source switch
  - app service restart
- Service/config operations never automatically re-enable command sending.
- High-risk flight controls use browser confirmation in the initial UI.
- YAML paths are server-side whitelisted; arbitrary filesystem access is not
  exposed.
- YAML is parsed before replacement; invalid input cannot overwrite a valid
  configuration file.

## 6. Service Management

Runtime restart actions use configurable commands from `config/app.yaml`.
Deployment should configure them through `systemd`, for example:

```yaml
services_control:
  restart_app_command: ["systemctl", "--user", "restart", "uav-app.service"]
  restart_yolo_command: ["systemctl", "--user", "restart", "uav-yolo.service"]
```

The app can reconnect telemetry internally because `LinkManager` is already an
application-owned service. YOLO and app process restarts are externally
supervised operations.

## 7. Implementation Sequence

1. Add Web UI configuration dataclasses and compatibility handling for the
   current terminal `ui_enabled` setting.
2. Add `web_ui` FastAPI backend, static front-end, audit logger, YAML store,
   and tests for path validation/save/restore.
3. Expose safe public operations and snapshots from `SystemRunner` and
   `ServiceManager`, including telemetry reconnect.
4. Add `yolo_app` annotated MJPEG stream module and configuration on RK3588.
5. Validate the common Web UI and RK3588-specific video integration.
6. Port the common change set and the PyTorch-side MJPEG hook into `main` and
   `platform/x86`.
7. Run tests and dry-run smoke checks for every updated branch.

## 8. Acceptance Checklist

- Web page loads with `python -m app.main` when Web UI is enabled.
- Terminal UI remains independently configurable.
- Annotated video is shown from `yolo_app`; targets can be locked through the
  picture and the list.
- All button and CLI actions use existing command dispatch paths.
- Mission discovery is dynamic.
- Operator actions persist across page reload and app restart.
- System state and event panes do not fill with repeated high-rate TX records.
- Approved YAML files save and restore one prior version.
- Active mission save-and-apply leaves `SEND=OFF`.
- Telemetry save-and-reconnect leaves `SEND=OFF`.
- YOLO and app restart endpoints execute configured supervisor commands.
- RK3588 and x86 offer the same UI features.
